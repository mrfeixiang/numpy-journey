import numpy as np

# 나의 첫 번째 파이썬 코드
# 2026년 6월, 손이 떨렸다
a = np.array([1, 2, 3])
print(a)
[1 2 3]

작동했다.
겨우 다섯 줄이었지만, 내가 쓴 코드가 실행되는 것을 본 순간, 심장이 두근거렸다. 이것이 프로그래밍이구나.

💥 첫 번째 에러 — TypeError

자신감이 붙어서 2차원 배열을 만들어보려 했다. "0으로 채워진 3×4 배열을 만들자!" 

np.zeros(3, 4)
TypeError
TypeError: Cannot interpret '4' as a data type

...뭐? 4가 왜 데이터 타입이야?

15분 동안 화면을 쳐다봤다. 구글에 "numpy zeros two arguments error"를 검색했다. Stack Overflow에서 답을 찾았다: shape는 튜플(tuple)로 넘겨야 한다. 

# ✗ 틀린 코드
np.zeros(3, 4)       # 3은 shape, 4는 dtype으로 해석됨

# ✓ 올바른 코드
np.zeros((3, 4))      # (3, 4)라는 튜플을 넘김

괄호가 두 겹이다. zeros(3, 4)가 아니라 zeros((3, 4)). 왜? 첫 번째 인수가 shape이고, shape는 튜플이니까.

🤯 (5,) — 이게 뭐야?

.shape를 찍어봤더니 이상한 게 나왔다. 

a = np.array([1, 2, 3, 4, 5])
print(a.shape)

# 출력: (5,)

(5,)? 쉼표 뒤에 아무것도 없는데? 에러인가?

15분 동안 구글링했다. 파이썬에서 원소가 하나인 튜플은 쉼표를 붙여야 한다는 걸 알았다. (5)는 그냥 숫자 5이고, (5,)는 "5라는 원소를 가진 튜플"이다.

📌 오늘 배운 것:
(5) → 정수 5 (괄호는 연산 우선순위)
(5,) → 길이 1인 튜플 (1차원 배열의 shape)

쉼표 하나에 15분. 이 속도로 세 파일을 합치려면 대체 며칠이 걸릴까?

🧭 axis=0 vs axis=1 — 세계에서 가장 혼란스러운 개념

np.concatenate()를 배우면서 axis 파라미터를 만났다. 

a = np.array([[1, 2],
              [3, 4]])

b = np.array([[5, 6],
              [7, 8]])

# axis=0으로 합치면?
np.concatenate([a, b], axis=0)
# [[1, 2],
#  [3, 4],
#  [5, 6],
#  [7, 8]]   ← 행이 늘어났다 (4×2)

# axis=1로 합치면?
np.concatenate([a, b], axis=1)
# [[1, 2, 5, 6],
#  [3, 4, 7, 8]]   ← 열이 늘어났다 (2×4)

axis=0이 "아래로"이고 axis=1이 "옆으로"? 아니, 잠깐, axis=0이 "행 방향"이라는 건 행이 늘어난다는 건가, 아니면 행을 따라서 간다는 건가?

머리가 빙빙 돌았다. 노트에 그림을 그리고, 지우고, 다시 그렸다. 창밖을 멍하니 바라봤다.

그러다가 Real Python 글에서 결정적인 단어를 발견했다:

"The axis that gets collapsed."
axis=0으로 연산하면, axis 0이 사라진다(접힌다). — Real Python, "NumPy axes explained"

유레카.

np.sum(a, axis=0)을 하면, axis 0 (행 방향)이 접혀서 사라지고, 결과는 열만 남는다. 그래서 (2,3) 배열에 axis=0 합계를 하면 shape가 (3,)이 된다 — 행 차원이 사라졌으니까!

🧠 axis 방향 다이어그램 
          axis=1 (열 방향 →)
          ───────────────→
    a = [[1, 2, 3],     |  axis=0
         [4, 5, 6]]     |  (행 방향 ↓)
                         ↓

  np.sum(a, axis=0)          np.sum(a, axis=1)
  = [5, 7, 9]               = [6, 15]
    ↑                          ↑
    axis 0이 접힘               axis 1이 접힘
    shape: (2,3) → (3,)        shape: (2,3) → (2,)

  np.concatenate(axis=0)     np.concatenate(axis=1)
  = axis 0 방향으로 쌓기       = axis 1 방향으로 붙이기
  = 행이 늘어남                = 열이 늘어남

이 다이어그램을 이해하는 데 거의 하루가 걸렸다. 하지만 이해한 순간, 세상이 달라 보였다. 배열은 더 이상 숫자 덩어리가 아니라, 방향과 차원을 가진 구조가 되었다.

💡 Day 1-2 핵심 교훈:
에러 메시지는 적이 아니다. 선생님이다.
"Cannot interpret '4' as a data type"이라는 에러가 나에게 튜플을 가르쳐주었다.

제 2 장

NaN — 자기 자신과 같지 않은 것

😱 🤯 💭 🌙

💔 세계관이 무너지는 순간

CSV 파일을 읽다 보면 빈 칸이 있다. NumPy에서는 이것이 NaN(Not a Number)으로 표현된다. 별거 아닌 줄 알았다.

그런데. 

x = np.nan

print(x == x)        # ???
False

뭐?!

x == x가 False? 자기 자신과 같지 않다고? 이건... 수학적으로 말이 안 되는데? 

print(np.nan == np.nan)    # False
print(np.nan != np.nan)    # True
print(np.nan > 0)         # False
print(np.nan < 0)         # False
print(np.nan + 1)         # nan

0보다 크지도 않고, 0보다 작지도 않고, 자기 자신과 같지도 않다. 유령 같은 존재. 어떤 연산을 해도 결과는 NaN. 전염병처럼 퍼진다.

📐 IEEE 754 — 그 뒤의 이유

구글링을 하다가 IEEE 754라는 부동소수점 표준을 알게 됐다. 1985년에 만들어진 이 표준이, "NaN은 자기 자신과 같지 않다"고 정의한 것이었다.

📌 왜 NaN ≠ NaN인가?

NaN은 "이 값은 의미 없음"을 뜻한다.
의미가 없는 두 값이 서로 같다고 하면,
0/0 == sqrt(-1)도 True가 된다.
그건 더 이상하다. 그래서 NaN은 아무것과도 같지 않다 — 자기 자신을 포함해서.

NaN을 찾으려면 ==가 아니라 특수한 함수를 써야 한다: 

# ✗ 이렇게 하면 안 된다
mask = (data == np.nan)      # 항상 False

# ✓ 이렇게 해야 한다
mask = np.isnan(data)        # NaN인 곳을 True로

그날 밤, 불을 끄고 누워서 생각했다.

자기 자신과 같지 않은 것.
존재하지만 비교할 수 없는 것.
어떤 것과 만나도 상대를 자기처럼 만들어버리는 것.

...묘하게 낭만적이다.

NaN은 프로그래밍에서 가장 철학적인 값일지도 모른다. "나는 존재하지만, 정의할 수 없다"는 선언. 부동소수점의 실존주의.
⚠️ Day 3 생존 팁:
데이터에 NaN이 있으면, np.nanmean(), np.nansum() 등 nan-safe 함수를 쓸 것. 아니면 NaN이 모든 계산을 삼켜버린다.

제 3 장

병합의 지옥, 그리고 달 착륙

😤 🔥 😭 🚀

🔗 pd.merge() — 진짜 보스 등장

Day 4. 드디어 진짜 과제에 착수했다. 세 파일을 합치는 것. NumPy만으로는 한계가 있어서, 교수님이 "병합 단계에서만" Pandas merge()를 허용하셨다. 하지만 왜 그것이 필요한지 이해한 다음에 쓰라고 하셨다.

Pandas의 pd.merge()는 SQL의 JOIN과 같다. 두 테이블을 공통 열(key)로 연결하는 것이다. 개념은 이해했다. 실행은 지옥이었다.

👻 rename()의 무음 실패

세 파일의 key 열 이름이 달랐다. Padel.csv에는 Name, Mutation.csv에는 CELL_LINE_NAME, Response.csv에는 COSMIC_ID. 이름을 통일해야 했다. 

# 열 이름 바꾸기 시도
padel_df.rename(columns={'Name': 'cell_line'})

# 바뀌었는지 확인
print(padel_df.columns[:3])
Index(['Name', 'nAcid', 'ALogP'], dtype='object')

...아직도 'Name'이잖아?!

에러도 안 나고, 경고도 안 나고, 그냥 조용히 아무것도 안 했다. 30분 동안 코드를 뚫어지게 쳐다봤다. 대체 뭐가 잘못된 거야?

구글에 "pandas rename not working" 검색. Stack Overflow 첫 번째 답변:

rename()은 기본적으로 새로운 DataFrame을 반환하고, 원본은 변경하지 않습니다. inplace=True를 쓰거나 결과를 다시 할당하세요. — Stack Overflow, 득표 1,247 
# ✗ 원본이 안 바뀜
padel_df.rename(columns={'Name': 'cell_line'})

# ✓ 방법 1: 다시 할당
padel_df = padel_df.rename(columns={'Name': 'cell_line'})

# ✓ 방법 2: inplace
padel_df.rename(columns={'Name': 'cell_line'}, inplace=True)
⚠️ 무음 실패(Silent Failure)는 에러보다 무섭다.
에러는 적어도 "틀렸다"고 알려준다. 무음 실패는 "괜찮다"고 속인다. 이것이 내가 NumPy의 명시성을 존경하게 된 이유다.

🔄 전치(Transpose) — 4단계의 깨달음

Mutation.csv의 shape는 (21213, 1002)였다. 행이 유전자이고, 열이 세포주. 하지만 병합하려면 행이 세포주여야 했다. 행과 열을 바꿔야 한다. 즉, 전치(Transpose).

나의 전치 이해 과정:

# 단계 1: "전치가 뭐야?" (Google: "numpy transpose")
# 행과 열을 바꾸는 것

# 단계 2: 작은 예제로 확인
m = np.array([[1, 2, 3],
              [4, 5, 6]])
print(m.shape)       # (2, 3)
print(m.T.shape)     # (3, 2)  ← 행과 열이 뒤집혔다!

# 단계 3: 실제 데이터에 적용
mutation_T = mutation_array.T
print(mutation_T.shape)   # (1002, 21213)

# 단계 4: 깨달음 💡
# 전치는 단순히 행열을 바꾸는 게 아니다
# "데이터를 다른 관점에서 보는 것"이다
# 유전자 중심 → 세포주 중심으로 시점 변환
✅ 전치의 본질:
같은 데이터, 다른 관점.
행렬의 전치는 "누구의 시점에서 볼 것인가"를 바꾸는 것이다.
유전자가 주인공이었던 테이블을, 세포주가 주인공인 테이블로.

🌙 Day 5 — 모든 것을 직접 타이핑하다

Day 5의 목표는 단순했다: 처음부터 끝까지, 복사-붙여넣기 없이, 모든 코드를 직접 타이핑하는 것.

구글을 참고하되, 코드를 보고 이해한 다음 창을 닫고, 내 손으로 치는 것. 한 줄 한 줄. 

# === 나의 코드, 나의 손으로 ===

# 1. 라이브러리
import numpy as np
import pandas as pd

# 2. 파일 읽기
padel = pd.read_csv('Padel.csv')
mutation = pd.read_csv('Mutation.csv')
response = pd.read_csv('Response.csv')

# 3. 전치 (유전자 → 세포주 중심)
mutation_T = mutation.set_index('Hugo_Symbol').T
mutation_T.index.name = 'cell_line'
mutation_T = mutation_T.reset_index()

# 4. 키 열 이름 통일
padel = padel.rename(columns={'Name': 'cell_line'})

# 5. 병합
merged = pd.merge(response, padel, on='cell_line', how='inner')
merged = pd.merge(merged, mutation_T, on='cell_line', how='inner')

# 6. 결과 확인
print(merged.shape)

이제는 12분 만에, 아무것도 보지 않고, 에러 없이 해냈다.

제 5 장

Nam 교수님의 철학 — 왜 Pandas 대신 NumPy인가

🎓 🧠 💭

7일의 여정이 끝나고 나서야, 교수님이 왜 그런 과제를 냈는지 이해하기 시작했다. Pandas를 금지한 것이 아니었다. Pandas를 이해 없이 쓰는 것을 금지한 것이었다.

이유 1: Pandas는 블랙박스, NumPy는 투명하다

Pandas의 pd.merge()는 마법처럼 작동한다. 입력을 넣으면 결과가 나온다. 하지만 그 안에서 무슨 일이 일어나는지 — 인덱스 정렬, 키 매칭, NaN 처리, dtype 변환 — 을 알 수 없다.

🔬 NumPy 방식

배열의 shape을 직접 확인한다.
axis를 명시적으로 지정한다.
전치를 수동으로 한다.
→ 매 단계가 보인다.

📦 Pandas 방식 (처음부터)

merge() 한 줄로 끝난다.
내부 동작을 모른다.
에러가 나면 대처할 수 없다.
→ 결과만 보인다.

이유 2: Shape 사고방식이 데이터 과학의 기초

NumPy를 쓰면 .shape를 끊임없이 확인하게 된다. "이 배열은 (217, 2326)이니까, 전치하면 (2326, 217)이 되고, concatenate하려면 행 수가 같아야 하니까..."

이 "shape으로 생각하는 습관"이야말로 데이터 과학에서 가장 중요한 기초 체력이다. TensorFlow든, PyTorch든, R이든 — 차원과 shape을 이해하지 못하면 아무것도 할 수 없다.

이유 3: 전치(Transpose)는 행렬 사고의 분수령

전치를 처음 만났을 때, 그것은 단순히 "행과 열을 바꾸는 것"이었다. 하지만 실제로 .T를 사용하면서 깨달았다 — 전치는 "데이터의 관점을 바꾸는 것"이다.

유전자 중심 데이터를 세포주 중심으로 바꾸는 것. 이것을 이해하는 사람과 이해하지 못하는 사람 사이에는 넘을 수 없는 벽이 있다. 교수님은 그 벽을 넘게 하려고 한 것이다.

이유 4: 디버깅 능력 > 코드 작성 능력

Pandas의 무음 실패(rename() 사건)가 나에게 가르쳐준 것: 코드를 쓰는 능력보다 에러를 찾는 능력이 더 중요하다.

NumPy는 에러를 명시적으로 던진다. TypeError, ValueError, AxisError. 이 에러들과 싸우면서 나는 디버깅하는 법을 배웠다. 에러 메시지를 읽는 법, 가설을 세우고 테스트하는 법, Stack Overflow에서 정확한 키워드로 검색하는 법.

이유 5: 도구는 바뀌지만, 행렬 논리는 변하지 않는다

2026년의 최신 라이브러리가 2030년에도 쓰일까? 아마 아닐 것이다. 하지만 행렬의 shape, axis, 전치, 브로드캐스팅 — 이런 개념은 1950년대부터 2050년까지 변하지 않는다.

교수님은 도구가 아니라 사고방식을 가르치려 한 것이다.

🚗 비유: 수동 변속기 먼저

자동 변속기 차를 운전하기 전에, 수동 변속기를 배우는 것과 같다.

수동 변속기를 배운 사람은 자동 변속기를 이해하면서 쓸 수 있다. 기어가 언제 바뀌는지, 엔진이 왜 그런 소리를 내는지, 오르막에서 왜 차가 밀리는지 안다.

자동 변속기만 배운 사람은 — 그냥 D에 넣고 달린다. 문제가 생기면 속수무책이다.

NumPy는 수동 변속기다. Pandas는 자동 변속기다.
수동을 먼저 배워야, 자동을 제대로 쓸 수 있다.

🎓 교수님의 말씀

과제를 제출하던 날, 교수님께서 이렇게 말씀하셨다:

"나는 NumPy나 Pandas를 가르치는 게 아닙니다. 모를 때 스스로 알아내는 방법을 가르치는 겁니다. 라이브러리는 바뀝니다. API는 바뀝니다. 하지만 '모르는 것을 만났을 때 당황하지 않는 능력' — 그건 평생 갑니다." — Nam 교수님

그제서야 이해했다. 7일 동안 내가 배운 것은 NumPy가 아니었다. 배우는 방법을 배운 것이었다.
